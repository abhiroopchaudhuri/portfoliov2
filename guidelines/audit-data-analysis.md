**Method note — "collapsed" vs "expanded" counts:** each row in the DEQUE JSON is already one test-run observation against one `unit` (component/page) for one `rule`. The DEQUE export itself groups DOM nodes inside the `source` array of a single row (e.g., "Close icon" found across three form fields appears as ELEMENT 1/2/3 inside one row). Therefore:
- **Collapsed count** = number of rows in the JSON (one rule × one component). This is what every "count" in this report shows unless otherwise noted.
- **Expanded count** = if we exploded every `<!--ELEMENT n-->` block in `source` to its own occurrence, totals would be higher. Spot-checks show 1.0–3.0 elements per row (typical = 1). A rough upper-bound multiplier is ~1.5×. All charts in the case study should use the collapsed (JSON row) count for accuracy — it is what DEQUE itself reports as "issue count".

**Data-source heuristic for MDS-attribution (used in §"Cross-product impact"):** an issue row is classified as *MDS-attributable* when its `source` HTML contains the string `DesignSystem-` (the `data-test` prefix generated by @innovaccer/design-system components). This is the most reliable deterministic signal available in the export. Rows without this signal are either:
- genuinely app-specific code (product-team-owned `<button class="edit__button">` etc.), or
- structural/page-level issues (page title, reading order, image alt text on product imagery, heading hierarchy) that DS cannot fix even when the component on the page is an MDS one.

### 1.5 Key themes in MDS v1

1. **Focus indicator contrast (1.4.11.b) — 40 rows, 22% of the file.** The focus ring outer color (#F8F8F8) against the default page background (#FFFFFF) measures 1.06:1, failing the 3:1 threshold. This single design-token bug in MDS propagates across every focusable component: buttons, inputs, selects, chips, dropdowns, links, vertical nav items, grid cells, stepper nodes. *This is the single biggest root-cause bug in the whole audit.*
2. **Programmatic label not descriptive (2.4.6.b) — 18 rows.** Icon-only buttons (close, visibility, edit, chevron, calendar) expose `aria-label` values like "close" or raw icon glyph text, not descriptive names like "Clear date-of-birth field" or "Show password".
3. **Label-in-name mismatch (2.5.3.a) — 14 + 5 = 19 rows.** Visible label differs from accessible name — a speech-input blocker.
4. **Link/text contrast failures (1.4.3.a) — 17 rows.** Subtle link color and caption text fail 4.5:1 on white.
5. **Name/Role/Value gaps (4.1.2.a) — 29 rows spread across 9 rules.** `<i role="button">` used instead of `<button>`; custom grids without `role="grid"`; selects without `aria-expanded/aria-selected`; missing form labels; sort state on table headers not exposed.
6. **Keyboard inaccessibility (2.1.1.a) — 6 rows.** Table cell resize handle, date-range drag, stepper nav keys not reachable.
7. **Reflow / resize (1.4.10 + 1.4.4) — 8 rows.** Content is lost/cut off at 200% zoom and 320 px width — most commonly Table, Stepper Form, forms.
8. **No Windows High-Contrast-Mode failures were flagged by DEQUE** in either MDS audit file (confirmed via text search — zero hits for "WHCM", "forced-colors", "High Contrast", "box-shadow"). The only WHCM mention in any source file is row 8 of the CSV tracker: `Buttons / Windows High Contrast Mode / Not Needed / Yes / / / / / AA - done`. So WHCM work is captured by our internal tracker but was not scoped into the DEQUE audit.

## §7 — Cross-product Impact (the key deliverable)

### 7.1 Attribution method

Each row in a product audit is classified as:

- **MDS-attributable** if the `source` HTML contains the string `DesignSystem-` (the MDS `data-test` namespace). Fixing this issue in MDS is expected to resolve the occurrence automatically when the consumer upgrades.
- **Product-team** otherwise (bespoke app markup, product-specific images, page-level structure, heading hierarchy, page titles, reading order of composed sections, etc.).

This is a deterministic, conservative classifier. It **undercounts** MDS-attribution because some MDS usages in the apps don't emit the `data-test` (older versions, custom wrappers), so the real MDS share is at least these numbers and likely higher.

## §8 — Recommended narrative for the case study

1. **Opening stat**: "Three Innovaccer products were audited by Deque in Oct–Nov 2025, returning **3,823 accessibility issue instances**. A parallel audit of the Masala Design System itself returned **193 issues** across **42 unique WCAG rules** and **14 MDS patterns**."
2. **Leverage stat**: "Because MDS powers those products, **~51% of the product-level issues (1,951 of 3,823) are resolvable by a single design-system release.** In Case & Care Management that share rises to **61.5%**."
3. **Root-cause stat**: "Ten MDS-level root causes account for over **1,200 issue instances** across the three products — the single biggest being a focus-ring color token that fails 3:1 contrast on white (46× in MDS, 262× across products)."
4. **Depth stat**: "The internal tracker covers **333 component-level tasks across 51 components**, addressing every one of the 42 DEQUE-identified rules plus **~30 additional themes** (Windows High Contrast Mode, AI components, screen-reader-only labels, live-region announcements) that axe did not catch."
5. **Shipped stat**: "Of those 333 tasks, **73 shipped fully (design + dev + release)** across three releases (02/25, 03/19, 03/30/2026), and **38 more shipped as part of v4.23.0** release notes — clustered into Buttons, Chips, Forms, Modal/Sidesheet focus management, Table, Calendar, DatePicker, Select, and AI components."
6. **What MDS cannot fix**: be honest — page titles (50), product images (153), hand-rolled ARIA in dashboards (75) stay with product teams.

## §9 — Reproducibility notes

All counts were derived via `jq` over the verbatim JSON arrays on disk. Each count above is either:
- a length-of-filter query (`[.[] | select(...)] | length`), or
- a `group_by` aggregation.

No sampling was used on any file — every issue row in every JSON was scanned. The one-and-only data-driven heuristic is the string-match on `DesignSystem-` in the `source` HTML for MDS attribution (§7.1), which is explicitly labelled as a conservative lower bound.

For the CSV, counts come from `Grep` line counts on fixed substrings ("Yes,Yes,Yes,", "Release Notes - v4.23.0", date strings, status labels). The 333-task figure is the raw 337 file rows minus 1 header and 3 trailing blank/note rows; no merging or interpolation was applied.
